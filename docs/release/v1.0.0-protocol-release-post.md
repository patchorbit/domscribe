## Framing notes (read before editing)

This release lands the same week Google I/O 2026 announced WebMCP and Chrome DevTools MCP, and Antigravity 2.0 + Composer 2.5 shipped. The post MUST land as a **complement at a different layer**, not a competitor:

- WebMCP, Chrome DevTools MCP, and Antigravity's desktop-agent surfaces operate at **runtime**, exposing what the browser already sees.
- RCP operates at **build time + framework boundary**, mapping the running pixel back to the **exact source location** (`SourcePosition` is required on every `resolve_*`/`query_*` response — that is the v1 contract).
- We do not compete with the runtime MCPs. We hand them the file and line they cannot derive from the DOM alone.

If a reviewer reads the draft and comes away thinking "Domscribe is taking on Google," the framing failed. Rewrite.

Avoid:

- "Better than WebMCP" / "alternative to Chrome DevTools MCP" / "the only build-time…"
- Vague "AI-powered" / "next-gen" / "revolutionary" puffery.
- Comparison tables that imply zero-sum competition with platform-level MCPs.

Prefer:

- "Complement to" / "different layer" / "what the runtime cannot see."
- Specific: "the IDE knows the file, the agent knows the pixel — RCP is the wire that connects them."
- Concrete: name the `SourcePosition`-required guarantee and the §7 stability policy explicitly.

## LinkedIn / blog (long-form)

**Title:** `@domscribe/protocol@1.0.0` — a build-time complement to runtime MCPs

Today we shipped `@domscribe/protocol@1.0.0` on npm. It is the typed wire contract for **RCP** — the Runtime Context Protocol — which Domscribe has been speaking against IDEs and coding agents since v0.1.

Two things happened in May 2026 that put a sharp edge on this release:

1. Google I/O announced **WebMCP** — an open standard for browser-based agents — and **Chrome DevTools MCP**, exposing DevTools state to agents at runtime.
2. **Antigravity 2.0** and **Composer 2.5** shipped, both reaching directly into IDE and OS surfaces.

If you were watching from the cheap seats, it could look like Domscribe sits in the same lane as these runtime-MCP surfaces. It doesn't.

**RCP operates at a different layer.**

Runtime MCPs answer: _what is the browser doing right now?_ They expose the DOM tree, network log, console state, and a few computed signals to an agent. They are excellent at this. We use them ourselves.

RCP answers a different question: _where in the source does this rendered pixel come from?_ When a user clicks the "Sign in" button in their running app, an agent reading RCP gets back not just `<button>Sign in</button>` from the DOM — it gets back `apps/web/src/components/AuthForm.tsx`, line 42, the component name, the wrapping HOCs, the live props snapshot, and the styling bindings. That is a **build-time mapping** plus a **framework-aware capture**, neither of which a runtime-only MCP can produce.

`@domscribe/protocol@1.0.0` makes that contract stable. The package's role in our stack:

- It is the typed wire schema (`ManifestEntry`, `RuntimeContext`, `Annotation`, the RFC 7807 error envelope) that an IDE, coding agent, or downstream tool depends on. No transitive dependency on Domscribe's relay or runtime is required.
- It is the SemVer signal. Stable fields are documented in `docs/rcp/v1.md` §5 and frozen for v1. Additive changes follow §7.2. Breaking changes require v2 with a written migration guide and at least one minor's deprecation window.
- It is the conformance target. Other implementations — not just ours — can claim "RCP v1 conformant" against this package.

Two specific commitments in v1.0.0 worth calling out:

1. **`SourcePosition` is required on every `resolve_*` and `query_*` response.** A v1 implementation that returns a manifest entry without a source location is non-conformant. This is the load-bearing differentiation against runtime-only MCPs and we made it a schema-level invariant rather than a convention.
2. **Canonical tool names use the underscore grammar** (`domscribe_resolve`, `domscribe_annotation_get`, …). The legacy dotted names (`domscribe.resolve`) are aliased through v1.x for migration, but the canonical form matches what stricter MCP clients (Windsurf, Anthropic's first-party servers, most of the community ecosystem) expect.

If you build a coding agent, an IDE plugin, or a design-review tool that needs to bridge a running UI back to source, depend directly on `@domscribe/protocol` — not on `@domscribe/core` or `@domscribe/relay`. The protocol package is the unit you integrate against; the relay and runtime are one reference implementation.

```bash
pnpm add @domscribe/protocol
# or
npm install @domscribe/protocol
```

Spec: `docs/rcp/v1.md` in the repository, with the stability policy in §7. Reference server: `@domscribe/relay`. Source: github.com/patchorbit/domscribe.

We are explicitly **not** competing with WebMCP, Chrome DevTools MCP, or Antigravity. RCP is the part of the stack that those surfaces cannot reach from where they sit. If you are building on any of them, ship Domscribe alongside — the runtime MCP tells your agent what the user is looking at; RCP tells your agent where in your codebase to change it.

— the Domscribe team at Patch Orbit
